# Databridge

A multi-source data viewer and reporting tool. Simulates pulling data from multiple source systems (sales, customers, inventory), storing everything in a central Postgres database, and presenting it through a REST API and Svelte dashboard.

**Stack:** FastAPI · Svelte · Postgres · Docker

---

## Prerequisites

- [Docker Desktop](https://www.docker.com/products/docker-desktop/)

That's it. No Python, Node, or anything else required locally.

---

## Getting Started

**1. Clone the repo**
```bash
git clone git@github.com:gretschduojet1/databridge.git
cd databridge
```

**2. Start the stack**
```bash
./start.sh
```

This creates a `.env` from `.env.example` if one doesn't exist, then starts Docker. The defaults work out of the box for local development.

> **Note:** Auto-creating `.env` is a convenience for this demo project. In a real environment you would provision secrets through your deployment platform (e.g. AWS Secrets Manager, environment variables set on the server) rather than deriving them from a file in the repo.

First run pulls images and installs dependencies — takes a few minutes. Subsequent starts are fast.

**3. Seed the database**

In a second terminal:
```bash
docker compose exec backend python seed.py
```

Inserts demo users, 200 customers, 15 products, and 500 orders with realistic sample data.

## Database

Four schemas, each simulating a distinct source system:

| Schema | Tables | Simulates |
|---|---|---|
| `customers` | `customers` | CRM |
| `sales` | `orders` | Order management |
| `inventory` | `products` | Warehouse system |
| `auth` | `users` | Identity / access |

To connect directly:
```bash
docker compose exec db psql -U databridge -d databridge
```

---

## Security

- Passwords hashed with bcrypt
- JWT authentication on all non-public endpoints (30 minute expiry)
- Role-based access: `admin` and `viewer`
- Login endpoint rate limited to 5 requests/minute
- CORS locked to the frontend origin

## Roadmap

**Phase 1 — Docker + Database + Seed Data** ✅
- Three-service Docker stack (db, backend, frontend) ✅
- Postgres schema with three simulated source systems ✅
- FastAPI with repository pattern, Pydantic DTOs, Protocol interfaces ✅
- Seed script generating realistic sample data ✅
- Test infrastructure with pytest ✅

**Phase 2 — Auth + API Completion** ✅
- JWT authentication with role-based access ✅
- Rate limiting on login ✅
- Svelte login page with protected app shell ✅
- Sorting, filtering, and pagination on all list endpoints ✅
- Aggregation endpoints (sales totals, inventory summary, revenue over time) ✅
- Alembic database migrations ✅

**Phase 3 — Svelte UI** ✅
- Dashboard with stat cards and charts ✅
- Table views with sorting, filtering, and server-side pagination ✅
- Excel export on every data view ✅

**Phase 4 — Search & Filtering**
- Full-text search across customers, products, and orders
- Evaluate a dedicated search service (Meilisearch, Typesense, or Elasticsearch) vs Postgres full-text search — the repository pattern makes swapping straightforward
- Enhanced filters on reports: date ranges, multi-select regions/categories, revenue thresholds
- Saved filter presets per user

**Phase 5 — Background Jobs**
- Event dispatch system for async work (e.g. report generation, data sync)
- Job queue (Celery + Redis) with status tracking
- Worker processes running independently of the API
- UI progress indicators for long-running jobs

**Phase 6 — AWS Deployment (Free Tier)**
- Containerized deployment to AWS using free-tier services
- RDS Postgres (db.t3.micro, free for 12 months) replaces the local db container
- ECS Fargate or EC2 t2.micro for the backend and frontend containers
- Secrets managed via AWS Secrets Manager or Parameter Store (replaces `.env`)
- ALB or CloudFront in front of the frontend
- GitHub Actions CI/CD pipeline: run tests on PR, deploy to AWS on merge to main
